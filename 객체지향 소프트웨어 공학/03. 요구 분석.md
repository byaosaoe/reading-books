# 01. 도메인 분석
## 1. 도메인 분석의 장점
- 도메인을 분석하려면 먼저 정보를 모아야 한다
- 정보는 보통 도메인 전문가와 관련 서적 또는 현재 사용되는 소프트웨어나 문서로부터 얻을 수 있다
1. 빠른 개발
2. 좋은 시스템
3. 확장 예견

## 2. 도메인 분석서 작성
### ◼ 개요
- 도메인의 이름과 도메인 분석을 하는 동기를 기술한다
- 동기를 적을 때는 특정 문제를 해결하기 위해 소프트웨어 시스템을 개발하거나 확장하려는 의도를 기술한다

### ◼ 용어
- 도메인에서 사용하는 전문용어나 일상용어의 의미를 정확히 정의한다

### ◼ 개괄적 지식
- 도메인 전문가가 알고 잇는 중요한 사실이나 규칙을 말한다

### ◼ 고객과 사용자
- 소프트웨어를 누가 구입할 것인지, 어떤 산업 분야에서 사용하는지 등을 기술한다

### ◼ 환경
- 사용하는 장비나 시스템에 대해 기술한다
- 새로 개발하거나 확장하는 시스템은 여기에 기술한 환경에 맞게 작동되어야 한다

### ◼ 작업과 수행 절차
- 일에 착수하면서 인력이 담당하는 작업의 리스트를 작성한다.

### ◼ 경쟁 소프트웨어
- 사용자나 고객에게 도움이 될 소프트웨어 가운데 이미 시장에서 판매되어 사용되는 기술의 장단점을 논한다

### ◼ 도메인과 조직의 유사도
- 고객의 조직과 다른 조직의 공통점과 차이점을 알아둔다.

# 03. 요구 추출
## 1. 요구의 정의
1. 요구는 문장이다
2. 시스템이 무엇을 하는가를 나타낸다
3. 관련자 모두가 동의한 것이다
4. 고객의 문제를 적절히 해결하기 위한 것이다

## 2. 요구의 유형
### ◼ 기능적 요구(functional requirement)
- 시스템이 외형적으로 나타내는 기능이나 성능 등의 동작을 말한다
- 예를 들어, 현금 인출기의 외형적 기능은 현금 인출, 잔금 조회, 계좌 이체, 현금 서비스 등이다

### ◼ 비기능적 요구(nonfunctional requirement)
- 기능이 아닌 성능이나 효율, 반응 시간, 제약 조건 등도 사용자의 요구에 해당되는데, 이를 비기능적 요구라 한다

### 2-1. 기능적 요구
- 입력: 시스템이 받아들이는 입력이 무엇이며 어떤 조건에서 입력되는지에 대한 것이다. 사용자와 다른 시스템으로부터 유입되는 데이터와 명령어가 모두 포함된다.
- 출력: 시스템이 생성하는 출력이 무엇이며 어떤 조건에서 생성되는지에 대한 것이다. 출력은 화면에 디스플레이될 수도 있고 인쇄될 수도 있다. 특수 입출력 장치나 클라이언트, 서버와 같은 다른 시스템에 전달될 수도 있다.
- 저장: 시스템이 어떤 데이터를 다른 시스템이 사용할 수 있도록 저장하는지를 말한다. 어떤 의미에서 다른 시스템에 대한 입력이 될 수 있는 특수한 형태의 출력으로 해석될 수 있다.
- 컴퓨팅: 시스템에서 어떤 계산이 이루어지는가를 찾아낸다. 계산 과정은 어떤 수준의 독자라도 이해할 수 있을 정도로 쉽게 작성되어야 한다. 예컨대 작업 결과가 계좌번호 오름차순으로 정렬되어야 한다는 식으로 작성하면 된다. 이 단계에서는 특정 알고리즘으로 사용한다고 명기하지 않는다.
- 타이밍과 동기화: 이는 모든 시스템에 해당되지 않지만 하드웨어 장치를 제어하고 리얼타임에 작업을 수행하는 시스템에서는 아주 중요하다. 통신 시스템이나 발전소•공장 제어 시스템, 자동자•비행기 제어 시스템 등이 여기에 해당한다.

### 2-2. 비기능적 요구
#### 📍 품질 특정
- 반응 시간: 대규모 자료를 처리하고 네트워크 사용이 많은 시스템에서는 사용자가 최소 시간 안에 피드백을 받을 수 있어야 한다. 매우 엄격한 리얼타임 시스템에서는 반응 시간에 대한 요구 사항을 비기능적 요구가 아니라 기능적 요구로 분류하기도 한다.
- 처리량: 시간이 많이 걸리는 계산 프로그램이나 클라이언트의 요청을 계속 받아 처리하는 서버 프로그램의 요구는 분당 트랜잭션 수와 같은 방법으로 처리량을 표시하는 것이 좋다.
- 자원 사용량: 메모리나 네트워크 주파수 대역과 같이 중요한 자원을 사용하는 시스템의 경우에는 시스템이 사용하는 최대 용량을 표시해야 한다. 그래야 하드웨어 업그레이드 등의 계획을 잡을 수 있다.
- 신뢰도: 신뢰도는 시스템을 고장 없이 사용할 수 있는 평균 시간으로 측정한다. 고장 사이의 평균 시간이나 주어진 시간 동안의 고장률로 측정할 수 있다. 또한 어떤 것을 고장으로 볼 것인지 명확한 정의가 필요하다.
- 가용성: 가용성은 서버가 사용자에게 반응할 수 있도록 항상 실행되고 준비된 상태에 있는 시간을 의미한다.
- 장애에서의 회복: 고장으로 인해 영향을 받는 최대 허용치로 나타낸다. 장애에서 회복되는 자세한 절차에 대한 정의는 기능적 요구이다.
- 유지보수와 확장의 허용: 시스템이 장래에 적응할 수 있도록 릴리스 후에 이루어지는 변경을 기록으로 잘 남겨야 한다.
- 재사용성의 허용: 시스템의 일정 부분이 재사용될 수 있도록 설계되어야 한다는 정량적인 제약 조건을 명시할 수 있다.

#### 📍 시스템의 환경과 기술
- 플랫폼: 개발할 소프트웨어가 어떤 하드웨어와 운영체제에서 실행될 것인지 명확히 해야 한다. 플랫폼에 대한 요구는 대체로 최소한의 기기 사양을 제시하여 그 이상의 기종에서 수행 가능해야 한다는 것을 명시한다. 플랫폼에 관한 요구는 앞에서 설명한 자원 사용량에 관한 요구와는 의미가 다르다. 자원 사용량은 시스템이 그 이상의 자원을 사용하지 않는다는 것을 의미하고, 플랫폼에 대한 제약은 적합하지 않은 자원에서는 수행되지 않는다는 것을 의미한다.
- 사용 기술: 프로그래밍 언어나 데이터베이스 시스템을 명시한다.

#### 📍 프로젝트 개발 방법에 대한 계획
- 사용하는 개발 프로세스(방법론): 소프트웨어 품질을 높이기 위해 특정 프로세스를 따를 것을 요구 문서에 명시하는 경우도 있다.
- 비용과 납기일: 가장 중요한 제약 조건이다. 요구 분석서보다는 시스템 개발 계약서나 별도의 프로젝트 계획서에 명시하는 경우가 많다.

|관찰|사용자의 업무를 관찰하며 기록|감추어진 문제를 잘 드러냄|

# 05. 사용 사례 분석
## 1. 사용 사례 분석 과정
- 액터 찾기, 사용 사례 찾기, 사용 사례 다이어그램 그리기의 세 단계

### 1-1. 액터 찾기
- 사용하는 사용자 클래스 또는 다른 시스템을 결정하는 일
- 이러한 사용자 클래스나 외부 시스템을 액터라 부른다
- 액터는 시스템에 대해 각기 다른 작업을 필요로 하는 주체이다
- 사용 사례 분석에서는 사람이 아니라 별개의 업무 역할을 발견하는 데 관심을 두고 살펴봐야 한다.

### 1-2. 사용 사례 찾기
- 각 액터가 시스템을 사용할 필요가 있는 작업을 결정하는 일
- 이러한 작업을 사용 사례라 부르는데, 이는 시스템이 사용되는 특정한 유형을 나타내는 것이다
- 다시 말해 시스템의 범위를 정하고 고객이 문제를 해결하기 위해 시스템을 사용할 때 액터가 사용할 사용 사례를 나열하는 작업이다

| 액터 | 사용 사례 |
|:---|:---|
|대출자|- 제목으로 검색<br/>- 저자로 검색<br/>- 주제로 검색<br/>- 다른 사람이 대출한 책을 예약<br/>- 대출자의 개인 정보와 대출한 책의 리스트를 체크|
|대출 직원|- 대출자가 책을 대출<br/>- 반납된 책을 확인<br/>- 책의 대출 기간을 연장<br/>- 지불된 연체료를 기록<br/>- 대출자를 추가<br/>- 대출자의 정보, 주소, 전화번호 등을 갱신|
|사서|- 새 책을 도서관 장서로 추가<br/>- 책을 도서관 장서에서 삭제<br/>- 책에 대해 시스템이 기록한 정보를 변경|
|회계 시스템<br/>(별개로 작동)|- 대출자가 지불한 연체료를 전달|

### 1-3. 사용 사례 다이어그램 그리기
- 사용 사례를 분석한 최종 결과는 다이어그램으로 그리고 각 사용 사례를 더 자세히 분할한다
- 모든 시스템이나 서브시스템에 대해 사용 사례 다이어그램을 만들 필요는 없다

## 2. 사용 사례 간 관계
- 사건 흐름의 경로가 너무 복잡하여 하나의 사용 사례로 기술하기 어려울 때는 별도의 사용 사례로 만드는 것이 좋다
- 이렇게 사용 사례를 별도로 만들더라도 밀접하게 관련이 있는 경우 확장, 상세화, 포함 관계를 사용하면 의미를 효과적으로 표현할 수 있다

### ◼ 확장 관계
- 선택적인 상호작용을 명시할 때 또는 예외적인 경우를 다룰 때 사용한다
- 예를 들면 액터가 파일에 접근하기 위해 파일 이름을 잘못 입력했을 때 어떤 예외적인 사건이 일어나는지를 나타낸다.
- 또한 파일을 찾기 위해 이름을 입력하지 않고 브라우징 윈도우를 이용할 때 어떤 상호작용이 발생하는지를 나타낼 수 있다
- 별도의 사용 사례를 확장하여 나타내면 기본 사용 사례가 간단해진다
- 사용 사례 확장을 정의할 때 중요한 점은 사용 사례의 처음부터 끝까지 모든 단계를 기술하고 예외 상황을 다루는 부분을 포함해야 한다는 것이다

### ◼ 상세화 관계
- 클래스의 서브클래스와 같은 의미로서 일반화된 사용 사례와 여러 유사 사용 사례의 관계를 나타낸다
- 상세화된 사용 사례는 간략한 사용 사례에 대해 더 상세한 사건 흐름을 제공한다

### ◼ 포함 관계
- 여러 사용 사례 사이에 공통적인 부분이 있을 때 이를 사용 사례의 일부로 분리하여 나타낼 수 있다
- 즉 여러 사용 사례가 일련의 액션을 공유하는 것이다
- 중복되는 상호작용을 여러 개의 사용 사례에 되풀이하는 것보다 특수한 사용 사례를 만들어서 이를 다른 사용 사례에 포함하는 것이 효과적이다
- 이렇게 하면 서브루틴을 호출하듯이 하위 수준의 작업을 나타내는 사용 사례가 된다

## 3. 사용 사레 작성의 실제
- 사용 사례는 액터가 시스템을 이용하여 주어진 작업을 완료하기 위해 수행한 일련의 액션, 이벤트를 의미한다
- 사용 사례를 분석하는 목적은 사용자나 다른 시스템이 특정 목적을 달성하기 위해 시스템과 어떻게 상호작용하는지를 모델링하는 것이다
- 사용 사례를 분석하는 것은 작업을 분석하는 것과 유사하다
- 작업 분석은 컴퓨터 이용에 관계없이 작업을 이해하고 효율성을 향상하는 데 사용되고 있다
- 사용 사례를 작성할 때 가장 중요한 것은 사용 사례의 이름, 액터의 액션과 그에 따른 시스템 반응 순서이다

> - 사용 사례 이름: 사용 사례에 대한 간단한 이름을 나타낸다
> - 액터: 사용 사례를 이용하는 사용자나 외부 시스템을 말한다. 예를 들면 도서관 시스템에서 책을 대출하려는 대출자나 사서에 해당된다
> - 목적: 액터가 얻으려는 목적을 설명한다. 예를 들면 도서관 시스템에서 책을 대출하는 목적은 빌려가서 책을 읽으려는 것이다
> - 시작 조건: 액터가 이 사용 사례를 구동하기 위해 충족되어야 할 조건, 즉 시스템의 상태를 기술한다. 예를 들면 책을 대출하기 위해서는 빌려갈 책이 선택되고 대출자가 낼 연체료가 없어야 한다
> - 관련 사용 사례: 사용 사례 사이의 연관성을 나타낸다. 예를 들면 확장, 포함, 일반화 등의 관계가 있다
> - 사건의 흐름: 액터가 취한 액션과 이에 대한 시스템의 반응을 시간 경과 순서로 적는다. 액션과 이에 대한 반응을 매칭하기 쉽게 다단으로 쓰면 보기에 좋다
> - 종료 조건: 사용 사례가 종료된 후 시스템이 어떤 상태가 되는지 설명한다

### ◼ 시스템 내부에 계산 내용을 작성하지 않아도 된다
- 예를 들어 현금인출기에서 돈을 찾는 사용 사례를 작성할 때, 카드를 넣고, 여러 메뉴가 제시되고, 버튼을 누르고, 현금과 카드를 받는 사건을 적으면 된다
- 현금인출기가 은행의 컴퓨터와 어떻게 통신하고 시스템이 현금인출기 사용 요금을 어떻게 부과하는지 등은 기술하지 않아도 된다

### ◼ 특정 사용자 인터페이스 설계와 관계없이 작성해야 한다
- '불러오기 버튼을 누른다'보다는 '불러오기 명령을 선택한다'가 더 바람직하다
- 마찬가지로 사용자가 파일 이름을 직접 입력하는지, 리스트에서 마우스로 선택하는지 명시할 필요가 없다

| 액터 측 액션 | 시스템 측 반응 |
|:---|:---|
|1. '불러오기' 명령 선택<br/>3. 파일 이름 명시<br/>4. 선택 확인|2. '파일 불러오기' 대화창 디스플레이<br/><br/>5. 디스플레이에서 대화창 삭제|

### ◼ 액터가 시스템과 상호작용한 액션만 포함해야 한다
- 사용 사례는 액터가 시스템과 상호작용한 액션만 포함해야 한다

### ◼ 사용 사례 간 관계는 접점을 표시하여 나타낸다
- 여러 개의 사용 사례가 서로 관계를 맺을 때는 먼저 관련된 사용 사례를 밝히고 이벤트의 흐름 속에 관련 접점을 표시한다

## 4. 사용 사례의 효과
- 시스템의 범위를 정하는데 도움이 된다. 시스템이 무엇을 할 수 있고 무엇을 할 수 없는지 정의한다
- 개발 프로세스를 계획할 때 사용된다. 사용 사례의 개수는 프로젝트 규모를 나타내는 좋은 지표이며, 개발 완료된 사용 사례의 비율로 개발 진척도를 나타낼 수 있다
- 요구를 발견하고 검토할 때 사용된다. 제안된 기능의 일부가 사용 사례에 표현되지 않았다면 이는 시스템에서 제외될 수 있다. 요구가 사용 사례로 표현되면 사용자가 더 잘 이해할 수 있다. 따라서 사용 사례는 고객과 개발자의 계약과 같이 쓰일 수 있다.
- 테스트 케이스를 정할 때 기초가 된다
- 사용자 매뉴얼 구성에 사용된다

# 06. 요구 문서화
- 경우에 따라서는 여러개의 요구 분석서를 작성하는데, 시스템의 각 부분을 따로 작성하거나 상세 수준에 따라 별도로 작성할 수도 있다
- 이렇게 분리해서 작성할 때는 서로 보완이 되도록 작성한다
- 필요하면 다른 문서를 참조하도록 소개하고, 문서 간에 중복되는 내용이 없도록 한다
- 문서 간에 중복되는 내용이 있으면 변경해야 할 때 한 부분은 고치고 다른 부분은 그대로 두어 모순이 생길 수 있다

## 1. 요구 분석서의 구조
### ◼ 시스템의 크기
- 대규모 시스템은 여러 가지 아유로 상세한 요구 문서가 필요하다. 규모가 커서 설명할 것이 많기 때문이다
- 또한 여러 팀이 각 파트를 담당할 수 있도록 서브시스템으로 분하랗여 다른 사람들이 어떤 일을 하고 서브시스템이 어떻게 상호 동작하는지 명확히 알 수 있다

### ◼ 다른 시스템에 대한 인터페이스 요구
- 소규모  시스템이라도 다른 시스템이나 서브시스템이 그 시스템이 서비스를 어떻게 사용하고 의사소통하는지 요구로 정리할 필요가 있다
- 예를 들어 클라이언트와 서버가 통신하는 프로토콜을 명세하지 않으면 협력하여 동작할 수 없다

### ◼ 요구 취합 단계
- 초기 단계에는 정확하고 상세한 요구를 많이 작성하는 것이 중요하지 않다. 잘못되어 다시 작성하는 경우 많은 노력이 들기 때문이다
- 이 단계에서는 시스템이나 서브시스템의 개요를 한두 문단으로 쓰는 것이 좋으며, 이렇게 간단히 표현한 요구에 관련자들이 모두 동의하면 상세한 사항을 추가해나간다
- 시스템의 첫 사이클에서는 아주 간결한 문장으로 정의한 요구를 대략적인 프로토타입으로 개발해보는 것이 적합하다
- 그다음에는 프로토타입을 시험적으로 사용하면서 좋은 것과 나쁜 것에 대한 정보를 알아낸다
- 이렇게 프로토타이핑을 반복하여 개발하는 것이 요구를 모으고 통합하는 과정이라고 할 수 있다

### ◼ 도메인 및 기술에 대한 경험 수준
- 대략적인 요구를 기초로 프로토타입을 개발하여 아이디어를 검증하고 보유 기술의 신뢰성을 검토한다
- 새로운 프로그래밍 언어, 클래스 라이브러르, 프레임워크, 알고리즘, 하드웨어, 데이터베이스 등의 보유 기술을 검토하는 것이다

### ◼ 잘못된 요구에 대한 비용
- 고장을 일으키면 안전이나 환경에 심각한 위험을 줄 수도 있는 시스템은 정확히 명세화되어야 한다

## 2. 요구 분석서의 내용
- 소프트웨어가 계속 진화하면서 기존의 시스템에 기능이 추가되면 요구 분석서에도 변겨오딘 내용을 업데이트한다
- 그러나 추가 작성이 반복되면 유지보수를 어렵게 만들 수 있다

> - 문제: 시스템이 해결하고자 하는 문제를 간단명료하게 기술한다
> - 배경 정보: 요구를 이해하는 데 도움이 되는 정보를 제공한다. 고려해야 할 중요한 이슈나 결정하는 데 근거가 된 사항을 기술할 수도 있다. 근거가 너무 길면 기능적 요구나 비기능적 요구에 직접 기술하여 장황하지 않게 한다
> - 환경 및 시스템 모델: 시스템이 수행될 배경 및 시스템이나 서브시스템의 개요를 기술한다. 이때 다이어그램을 사용하는 것이 효과적이다. 시스템이 수행될 하드웨어를 명시하고 상호작용하는 다른 서브시스템이나 소프트웨어를 나타내야 하낟
> - 기능적 요구: 앞에서 언급된 대로 사용자나 다른 시스템에 제공될 서비스를 명시한다. 입력, 출력, 계산 기능과 시간 제약 등을 기술한다. 기능적 요구를 나타낼 때는 다이어그램이 유용하다
> - 비기능적 요구: 앞에서 언급한 대로 시스템 설계에 부여되는 제약 사항에 대해 기술한다
